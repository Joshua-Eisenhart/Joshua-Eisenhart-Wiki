# Current Docs vs Legacy Framing

## Why this routing page exists
The wiki now contains two different reading modes: current-docs-aligned material that should be treated as the preferred lower-entropy front-door cluster, and older legacy material that is useful but often inconsistent, speculative, or framing-dependent.

## Side-by-side
| Dimension | Current docs framing | Legacy framing |
|---|---|---|
| Authority | Preferred first, lower-entropy front door | Candidate/provenance-first |
| Style | Concrete, constraint-first, operational | Mixed, interpretive, speculative, historical |
| Use case | Reconciliation, instruction surface, LLM harness | Source history, alternative wording, older theory |
| Subclusters | current-docs-map, current-architecture-core, current-formal-methods-core, current-research-overlays, docs-alignment-catalog | legacy-source-history, legacy-speculative-frameworks, legacy-physics-cosmology, legacy-psychology-personality, legacy-governance-bootpacks, legacy-speculative-theory |
| Conflict handling | Wins when pages disagree | Preserved, but not treated as default truth |
| Reading order | First pass | Second pass |

## Role in the live wiki cluster
- **Strongest use:** routing/comparison page for deciding whether a question should stay in the current lane or detour into legacy provenance/routing first.
- **Weak use:** not the page for detailed legacy branch history, book-level comparison, or doctrine extraction inside any one legacy stream.
- **Authority boundary:** routing/comparison surface only. It decides reading mode and conflict posture; it does not itself promote legacy material to current authority.

## Recommended reading order
1. Start here when the question is whether a claim belongs to the current spine or a legacy branch.
2. Continue to [[legacy-source-history]] for provenance strata.
3. Continue to [[legacy-speculative-frameworks]] for branch grouping.
4. Read [[dark-empress-vs-grandmaster]] only if the branch question is specifically about the two major legacy books.
5. Use [[legacy-speculative-theory]] only when you need the warning-label page for fusion-heavy legacy synthesis.
6. Hand forward into the specific branch or current bridge page only after that routing decision is made.

## Practical rule
- `proven`, `candidate`, `open`, and `snapshot-labeled` are claim-status labels separate from style or authority. A page can be current/front-door without thereby making every claim on it canonical.
- If a current page and a legacy page disagree, use the current page as the default lower-entropy anchor for interpretation, then check whether the legacy page is carrying provenance, alternative framing, or unresolved residue worth preserving.
- If a legacy page adds provenance or a useful historical route, keep it linked but clearly labeled as candidate/history unless stronger proof exists.
- If the page is meant for LLM consumption, prefer current-docs-aligned phrasing.

## Legacy-cluster reading order
For the overlapping legacy cluster, the cleaner reading order is:
1. [[legacy-source-history]] for provenance strata
2. [[legacy-speculative-frameworks]] for branch-level grouping
3. the specific branch page you need:
   - [[leviathan-framework]] for governance/social OS

   - [[grandmaster-of-the-universe]] and [[the-dark-empress]] for major legacy books
   - [[emotional-evolution-personality-system]] for the personality/engine-grammar lane
4. current synthesis pages only after that, when translating the legacy material into live wiki language

## Leviathan v3.2 Position in the Legacy Map

Leviathan v3.2 is the primary legacy governance-OS document. Its framing is distinct from the physics/entropy legacy material and should be routed separately:

| Dimension | v3.2 framing | Relation to current docs |
|---|---|---|
| Primary domain | Governance, social OS, cooperative economics | Genealogical upstream — not front-door doctrine |
| Vocabulary | SBC/CRC/ECC, DPV, Paladin, Ugly Mirror, LVT | Not used in current constraint-first framing; traces governance vocabulary |
| Platform framing | Leviathan OS — intent-over-instruction semantic computing | Shares "intent understanding" pattern with current runtime; different vocabulary |
| Personality layer | Three-axis attractiveness, Four Fs, Mirror scores | Operationalizes [[emotional-evolution-personality-system]] at OS level |
| Coordination failure | Veto mechanism, rival Leviathans, finite resource principle | Genealogy for [[moloch-trap-reference]] |
| Reading position | Legacy governance subcategory | Read after current spine; entry via [[legacy-governance-bootpacks]] |

The Leviathan OS semantic computing platform (intent-over-instruction, self-organizing adaptive systems) is genealogically upstream of the current-docs runtime framing. Both treat intent-understanding as primary over instruction-execution, but the current system expresses this through constraint-first nominalist language while v3.2 uses natural-language AI and blockchain transparency. These are different framings of a structurally similar intent; they do not contradict.

For full v3.2 coverage see [[leviathan-framework]].
