# Viability vs Attractor

## Role in the live wiki cluster
- Strongest use: compact comparison page for when the question is whether a constraint-first system should be organized by survivable state-space regions or by convergence-to-set language.
- Weak use: first-stop authority page for current repo truth, runtime controller status, or broader dynamical-systems doctrine.
- Authority boundary: comparison/router page that helps readers keep viability language distinct from attractor language inside the current constraint-first stack; it does not replace the `hermes-current/` front door, the repo-current controller docs, or the narrower reference pages that define either family in more detail.

## What is being compared
Two different questions about a dynamical system:
- Can it stay within constraints?
- Where does it converge?

## Comparison
| Dimension | Viability kernel | Attractor basin |
|---|---|---|
| Core question | where can the system survive? | where does it go? |
| Requirement | at least one trajectory stays in K | trajectories converge to a set |
| Constraint role | central | often external or secondary |
| Control | explicit | usually autonomous |
| Failure mode | kernel can be empty | attractors often still exist |

## Verdict
For the Leviathan stack, viability is the better primitive because the project is constraint-first and survivorship-oriented, not convergence-first.
