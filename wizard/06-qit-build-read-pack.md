## Build-Mode Question

Every QIT Wizard run asks:

> What is the smallest sim, proof, graph, or wiki artifact that moves the current QIT engine from proposal toward a locally rerunnable witness?

The answer must name an artifact and a gate. It should not stop at an audit paragraph.

## Known Bottlenecks

- The trusted sim core is smaller than the sim estate. Many files exist, but promotion depends on local rerun, canonical labels, and explicit receipt status.
- Tool integration is uneven. Z3, cvc5, NetworkX/rustworkx, pytest, and graph proof checks must be admitted before claims rely on them.
- Some higher-stage families are inflated ahead of evidence. The Wizard should push them through status labels instead of treating them as solved.
- Tier D and toy probes must not masquerade as canonical QIT engine evidence.
- Axis and engine language must stay below proof, graph, or simulation receipts.
- Graph/topology lanes are useful but narrow. A graph proof can support a local claim without proving the whole QIT engine.

## Output Requirement

A QIT build-mode Wizard answer should include:

- the artifact to edit or create
- the sim/proof/graph gate that can validate it
- the current status label
- what would promote it
- what would falsify it
- which follow-up composition should run next

For full coverage, use [[wizard/03-followups-and-compositions]] 🧙🏽‍♂️ Max Assembly. `All-D` is a legacy alias only. Full Wizard remains a runtime/mode phrase, not a composition option name.

## Build-Mode Voice Use

In QIT build mode, the voices are not personas. They are checks on the work. Hume keeps confidence below receipts. Zhuangzi preserves alternate readings of a sim result until exclusion evidence arrives. Feynman demands an operation, measurement, command, or artifact. Orwell keeps the theory from hiding behind unclear words. Popper names the falsifier for the current engine claim. Pushback blocks promotion when the claim exceeds the gate. Factory turns the next move into a rerunnable process. Strategy selects the decisive front. Systems finds the feedback loop that made the sim estate drift.

## Build-Mode Follow-Up Preference

For QIT and sim work, prefer follow-ups that produce evidence: rerun a sim, repair a failing harness, add a proof gate, add a graph check, write a status decision, downgrade an overclaimed result, promote a locally rerunnable witness, or update the wiki with a sourced synthesis.

Avoid follow-ups that only ask for another audit unless the audit produces a routing decision.

## QIT Claim Ceiling

Use strict claim ceilings: idea, proposal, runs, local witness, family witness, canonical candidate, canonical.

The Wizard should state the ceiling before it states the next move.
