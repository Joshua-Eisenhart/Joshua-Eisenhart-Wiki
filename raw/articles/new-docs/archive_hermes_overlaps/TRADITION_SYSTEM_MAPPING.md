# Tradition → System Mapping

Date: 2026-04-05
Status: Mapping doc — connects external traditions to system claims without collapsing them into the system

---

## Purpose

The system has many external support traditions. This doc maps each tradition to the system claims it can support, and where it should not be overused.

The rule: support tradition is not root ontology.

---

## High-level mapping

### Formal methods
Best support for:
- admissibility gates
- witnesses and certificates
- fail-closed checking
- rejected candidate handling
- invariant discipline

Do not use as:
- ontology of the universe
- substitute for the system’s constraint substrate

### Differential geometry / bundles
Best support for:
- geometry buildup
- transport and connection language
- carrier layers
- spin structure / spinorial claims
- Hopf-fibration-style layering

Do not use as:
- proof that the system’s ontology is “just geometry”

### Quantum information / quantum Shannon
Best support for:
- cut families
- bridge/channel language
- density operators
- entropy / coherent-information language
- witness language for entanglement and correlation structure

Do not use as:
- a blanket metaphor for all correlation

### Information geometry
Best support for:
- distinguishability
- statistical manifolds
- metric structure on families of admissible states

Do not use as:
- a replacement for the core constraint grammar

### Active inference / predictive processing
Best support for:
- candidate updating
- salience and error correction
- bounded agents under uncertainty

Do not use as:
- root ontology unless explicitly proven useful

### Concurrency / trace theory
Best support for:
- order-sensitive composition
- irreversible passes
- append-only trace discipline
- noncommuting process structure

Do not use as:
- a substitute for the formal admissibility gate

### Topos theory
Best support for:
- contextual admissibility
- local truth values
- non-Boolean structure

Do not use as:
- a universal replacement for the system’s other foundations

---

## System-side mapping language

Use these exact mappings when writing system docs:
- admissibility charter -> formal acceptance conditions
- witness discipline -> proof/certificate discipline
- geometry buildup -> induced geometry / bundle construction
- bridge family -> transport or channel-like correspondence
- cut family -> bipartite partition or submanifold partition, if defined
- kernel family -> functional or operator on admissible states
- surviving candidate -> accepted candidate / verified candidate / admissible candidate

---

## Warnings

- Do not collapse all support traditions into one narrative.
- Do not let a support tradition become the ontology itself.
- Do not mix metaphor with proof language.
- Do not use one tradition’s jargon where an exact term already exists.

---

## Recommended use

This doc should sit between the reference docs and the system docs:
- reference docs explain each tradition precisely
- this mapping doc says how each tradition supports the system
- system docs state the actual candidate thesis and governing constraints

That separation helps keep the stack clean.
