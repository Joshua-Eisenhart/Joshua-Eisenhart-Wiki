# Leviathan World Engine Memo

## Overview
The Leviathan World Engine is a CS-facing name for the combined architecture formed by:
- Leviathan as control/governance layer
- QIT engines as nonclassical engine layer
- the holodeck as internal world-model layer
- an FEP-compatible adaptive loop as self-maintenance discipline

The goal is not to present a mystical or universal theory. The goal is to explain an architecture for bounded intelligence where lawful state, internal world construction, and strict process discipline live in one stack.

## Core claim
This project is not trying to build just another agent framework or workflow shell. It is trying to build a system where:
- intelligence operates over maintained internal world-state
- cyclic processes are only accepted when the underlying state geometry supports them
- outputs are admitted by bounded simulation, audit, and ordered construction rather than by nearby plausibility

In short: the system is trying to make engines, not just loops, out of structured information geometry.

## The four-layer stack
### 1. Leviathan
Leviathan is the control plane.
It provides:
- governance
- contracts
- gating
- audit receipts
- hierarchy/coordination rules
- mirror/accountability structure

Its job is to decide what is allowed to run, what counts as evidence, and what is admissible for promotion.

### 2. QIT engines
The QIT engine layer is what makes the architecture unusual.

A QIT engine is not just a process loop over arbitrary states. It is a cyclic process whose validity depends on the underlying admissible information geometry.

That means the cycle must respect:
- admissible state objects
- probe-relative distinguishability
- lawful transforms
- order-sensitive composition
- chirality/asymmetry where relevant
- coupling/coexistence constraints
- stricter admissibility as the ratchet tightens

Without that geometry, the engine collapses into an ordinary classical control loop with decorative language.

### 3. Holodeck
The holodeck is the internal world-model layer.
It provides:
- predictive projection
- internal world continuity
- world-coupled memory
- counterfactual testing space
- a place for simulated trajectories before irreversible action

This gives the engine an internal world to run over rather than only a prompt or a state vector.

### 4. FEP-compatible loop
An FEP-compatible loop is the adaptive maintenance layer.
It provides a disciplined cycle of:
- observe
- update model
- compare prediction with input
- choose action
- preserve viability/coherence

The safe claim is compatibility, not proof: the architecture is aligned with prediction-first, bounded self-maintenance logic.

## Why the QIT engine layer is special
The QIT engine layer is the special part.

Most systems treat an engine as a process that transforms state.
This project treats an engine as a constrained cycle whose reality depends on the geometry and distinguishability structure of the state space itself.

A normal loop is valid if:
- the code runs
- the transitions compose
- the metrics improve

A QIT engine is valid only if:
- the states are admissible
- the transforms are lawful
- relevant distinctions survive probe
- order-sensitive composition is preserved
- geometry supports the cycle
- coupling with other layers does not destroy it
- apparent gain is not paid for by hidden structural loss

That is why the exotic geometry is load-bearing, not optional.

## Why geometry first
The build order starts with geometry because geometry is where the engine's reality conditions become definable and testable.

Geometry-first means:
1. define bounded geometry legos
2. simulate them deeply under the QIT constraints
3. integrate the relevant tools where they are actually load-bearing
4. test bounded connections between geometry legos
5. only then test coexistence/nesting/layering
6. only then build broader integrated simulations

This is not arbitrary sequencing. It is the condition for making engine claims honest.

## The process rule that makes the system work
In this system, order is part of correctness.

The same root constraints that govern the modeled system also govern the builder and the LLM:
- F01 -> finite, bounded, witness-backed packets
- N01 -> exact order preservation, no opportunistic reordering

That means:
- bounded legos first
- bounded connections second
- no widening because something nearby is useful
- out-of-order work is failure, even when locally productive

This is why the wiki is being used as an LLM harness rather than just a knowledge base.

## What the system is for
The larger goal is not to replace human beings with abstract machine reason.
It is to build a system that can:
- maintain world models
- simulate before acting
- preserve lawful structure under tighter constraints
- support cultivated, auditable intelligence rather than improvisational drift

That makes Leviathan both:
- a science framework for recursive empirical construction
- an AI alignment framework for constrained participation and ordered cooperation

In the deeper integrated reading now emerging across the wiki, the recursive science method is not separate from the engine architecture. Leviathan's mirrors/accounting method, the holodeck's prediction-first perception/memory model, FEP-style update logic, and the QIT engine program are all treated as different surfaces of one recursive engine pattern.

## One-sentence summary
Leviathan World Engine is a constraint-first architecture where Leviathan governs behavior, QIT engines define lawful nonclassical cycles, the holodeck provides internal world-state, and an FEP-compatible loop maintains adaptive coherence under strict bounded process discipline.
