# Terrain Laws and Loop Geometry

## Definition
This page explains the current terrain/loop layer in cleaned mathematical terms.

The terrain families are not primitive metaphysical objects. They are structured one-parameter operator/channel families acting on the admitted qubit carrier and placed on different loop types generated by the Hopf/Weyl geometry. The loop geometry determines whether density is stationary or traversing, and the terrain/operator placement determines what kind of state change can be expressed on that loop.

## Key structures
### Carrier and spinor setting
The admitted local carrier is the qubit carrier `H = C^2` together with density states `D(C^2)` and the pure spinor carrier `S^3`.

A normalized spinor is written in Hopf coordinates as:
- `psi(phi, chi; eta)` on `S^3`
- Hopf projection `pi : S^3 -> S^2` gives the Bloch-vector image

The left/right Weyl split then introduces two sheets with opposite Hamiltonian sign:
- `H_L = +H_0`
- `H_R = -H_0`

This is why loop placement and chirality cannot be collapsed into a single flat operator picture.

### Pauli and ladder operators
The local algebra is generated by the Pauli basis:
- `sigma_x`
- `sigma_y`
- `sigma_z`

with the raising/lowering operators:
- `sigma_+ = (sigma_x + i sigma_y)/2`
- `sigma_- = (sigma_x - i sigma_y)/2`

These are the basic local operator ingredients from which the terrain laws are built.

### Connection and loop geometry
The cleaned geometry side is:
- fiber loop: density-stationary
- base loop: density-traversing

In the current full-math framing:
- fiber loop changes phase without changing the density state
- base loop changes the off-diagonal phase and therefore changes distinguishability

This distinction is critical because terrain placement on a loop is not arbitrary. A law acting on a density-stationary loop is not the same object as the same law acting on a density-traversing loop.

## Terrain families in current mathematical language
The current docs separate two layers that older wording often blurred together:
- intrinsic local operator families: `Ti`, `Te`, `Fi`, `Fe`
- later terrain families / placement families: `Se`, `Ne`, `Ni`, `Si`

So this page should not be read as saying that the terrain names replace the operator packet. The cleaner current reading is:
- `Ti`, `Te`, `Fi`, `Fe` are the admitted intrinsic local operator families
- `Se`, `Ne`, `Ni`, `Si` are the later terrain-family classification used when those operators are placed on Weyl sheets and loop contexts

At the registry level, `terrain_family_fourfold` is still `not_normalized_yet`, so terrain language remains a useful current framing layer but not yet a standalone fully-landed lego.

The current docs describe four recurring terrain-family classes on the local qubit carrier, but the registry still treats this packet as `not_normalized_yet`:
- `Se` — expansion / raising Lindblad family
- `Ne` — circulation / noncommuting Hamiltonian family
- `Ni` — contraction / lowering Lindblad family
- `Si` — stratified / commuting Hamiltonian family

In math-first language these are better read as:
- open dissipative outward family
- closed circulation family
- open dissipative inward family
- closed stratified family

The point is not their older symbolic names by themselves. The point is that they define four distinct terrain-family classes on the same carrier once the operator packet, Weyl sheet, and loop context are kept separate.

## Loops and placements
### Four loops
With left/right Weyl sheets and fiber/base loop geometry, the current terrain picture yields four loop placements at the sheet/loop level:
- left-fiber
- left-base
- right-fiber
- right-base

### Sixteen placements
Placing four terrain families across left/right sheets and fiber/base loops yields sixteen placements.
This is the first place where terrain/operator/loop assignment becomes an actual structural placement law rather than a list of operator names.

That matters because the system is trying to discover which placements are lawful, distinguishable, stable, and later reusable in engine cycles.

Hardening caution:
the sixteen placements are currently best read as a structured design space, not as sixteen already-admitted local objects. The repo still treats `placement_law` and related terrain-loop machinery as scattered and not yet normalized into a clean standalone probe.

## Relevance to this system
This page matters because the terrain/loop layer sits between the lower geometry spine and later engine promotion.

The current build order implies:
1. carrier and geometry first
2. then Weyl/chirality
3. then operator families
4. then loop placement and order sensitivity
5. only later: broader engine claims

So the terrain laws are not an early free-standing doctrine. They are a mid-layer construction surface where geometry, chirality, operators, and loop semantics are first forced to interact.

## Key results already implied by current docs
- Fiber/base loop distinction is load-bearing: one preserves density, one traverses density.
- The intrinsic operator packet and the terrain-family packet should not be collapsed: `Ti/Te/Fi/Fe` are the local operator families, while `Se/Ne/Ni/Si` are the later terrain-family classification used for placements.
- Left/right Weyl split changes the placement story because Hamiltonian sign and sheet ownership differ.
- Placement law is therefore a real mid-layer object, not mere annotation.
- The terrain packet is still weaker than the lower operator and loop legos in the current registry, so engine promotion should stay blocked behind cleaner normalization.
