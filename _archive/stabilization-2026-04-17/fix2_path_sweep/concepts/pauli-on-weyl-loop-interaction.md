# Pauli On Weyl Loop Interaction

## Definition
This page isolates one of the most important still-partial layers in the current build: Pauli/operator action on the left/right Weyl layer together with fiber/base loop geometry.

This is not yet a finished or closed doctrine page.
It is a current-docs-aligned interaction page describing the layer where:
- local operator math
- left/right Weyl chirality
- loop geometry
- and placement/order sensitivity

first become coupled in a way that matters for later engine construction.

This page names a dependency packet rather than a single closed lego. The local basis/asymmetry anchors are now real, but the broader operator-family and loop-placement layers are still separate and should not be collapsed together.

## Why this layer matters
The system does not just need:
- Weyl spinors
or
- Pauli operators
or
- loop geometry

in isolation.

It needs to know what happens when these layers are placed on each other.
That is the point where the manifold starts behaving like a constrained runtime rather than like separate local math notes.

In practical terms, this layer is where the build moves from:
- local geometry objects
- local operator objects

toward:
- lawful local engine pieces
- order-sensitive placements
- the preconditions for later transport/differential/emergence stories

## Ingredients
### 1. Weyl split
Current full-math framing gives:
- `H_L = +H_0`
- `H_R = -H_0`

So the same geometric point can carry a left and right realization with opposite Hamiltonian sign.
This means operator action cannot be treated as chirality-neutral by default.

### 2. Pauli basis
The local operator basis is generated by:
- `sigma_x`
- `sigma_y`
- `sigma_z`

with ladder operators:
- `sigma_+`
- `sigma_-`

These define the current local dephasing and rotation families.

### 3. Loop grammar
Loop geometry distinguishes:
- fiber loop — density-stationary
- base loop — density-traversing

That means the same operator family may have very different observational consequences depending on whether it is placed on a fiber or base loop.

## What interaction means here
Pauli-on-Weyl-loop interaction means:
- take the local operator packet
- place it on left/right Weyl sheets
- place that on fiber/base loop contexts
- check which differences are real under admissible probes

So the interaction layer is not one object.
It is a structured family of local placements and order-sensitive comparisons.

## Immediate local questions the layer asks
- Does the same Pauli action behave differently on left vs right Weyl sheets?
- Which changes are visible only on density-traversing loops?
- Which distinctions are phase-only and therefore not density-visible?
- Which operator/loop pairings preserve or destroy distinguishability?
- Which local asymmetries are strong enough to survive into later engine placement law?

## Why this should come before flux
A key current correction is that flux should not be treated as an early local object.

A left/right Weyl spinor by itself is too thin.
Pauli action by itself is too thin.
Loop geometry by itself is too thin.

Flux, if it is honest, likely requires several of these layers to run together first.
So Pauli-on-Weyl-loop interaction is part of the dependency chain that should exist before flux candidates are taken seriously.

## Relation to engine construction
This is one of the last clearly local layers before broad engine promotion starts to become tempting.

It matters because a later engine cycle needs lawful pieces such as:
- left/right placement-sensitive transforms
- loop-sensitive transforms
- order-sensitive compositions
- local asymmetries that survive probe checks

Without those, engine language becomes premature.

- `composition_order_sensitivity` is the local operator/channel row already useful here
- `composition_order_noncommutation` is the later engine/placement row in the repo backlog

This page should use the first directly and mention the second only as a downstream follow-on.

## Relation to the CS framing
In CS/dev terms, this layer is the first place where the runtime legality of transforms becomes context-sensitive in a richer way.

The same local operator is not “the same operation” in every context if:
- chirality differs
- loop type differs
- density visibility differs
- composition order differs

So this page is really about context-sensitive operator semantics on a constrained runtime manifold.
