# Harness Translated Companion

Proof-of-method page for the harness layer. This page explains how the translation stack is supposed to operate on a real canonical page, using [[constraint-surface-translated]] as the worked companion to [[constraint-surface-and-process]].

## Purpose

The harness layer needs one concrete demonstration page, not just rules in the abstract. The demonstration must show:
- one authoritative source page
- one translated companion page
- explicit rule application
- controller-facing reasons the translation matters

In this wiki, the canonical worked example is:
- source page: [[constraint-surface-and-process]]
- translated companion: [[constraint-surface-translated]]

## What the worked example proves

### 1. Translation is not paraphrase

The companion page does not merely simplify prose. It restates the same claims in multiple controlled framings:
- ordinary
- nominalist
- systems
- executable

That proves the harness can preserve content while changing the framing discipline.

### 2. Constraint-first language can be operationalized

The companion forces claims to name:
- active constraints
- role/process boundaries
- executable checks
- admissibility versus exclusion structure

That is the core harness move: it prevents drift back into object-first or universal language.

### 3. Controller prompts can demand translated reporting

Because the worked example exists, a controller can say:
- report in nominalist form
- give the executable restatement
- separate ordinary framing from systems framing

Without this page pair, those instructions would stay vague.

## Rule application map

The worked companion exercises the harness rules in a way a controller can inspect.

### Translation rules in play

- [[nominalist-translation-rules]] Rule 1: object -> survivor
- Rule 2: creation -> selection
- Rule 4: causation -> constraint coupling
- Rule 5: truth -> admissibility
- Rule 7: universal -> context-bound

### Bias inversions in play

- [[harness-bias-inversions]] / [[llm-bias-inversion-rules]]
- reification -> constraint-bounded survivor language
- narrative smoothing -> contradiction preservation
- universal framing -> context-bound framing
- compression -> explicit expansion

## How to use this page in practice

### For controller work

When a worker returns a broad conceptual summary, ask:
1. what is the source page?
2. where is the translated companion?
3. which framing is being used right now?
4. what executable or status-bound restatement exists?

If those answers are missing, the summary is not yet harness-clean.

### For wiki work

When adding a high-importance concept page:
1. decide whether it needs a translated companion
2. if yes, preserve the authoritative page and create a separate translated page
3. cross-link both pages immediately
4. update the index and log in the same batch

### For sim/result interpretation

Use the companion pattern when a page is especially vulnerable to:
- Platonic language
- status collapse
- causal overstatement
- geometry/axis conflation

## Why this matters for the harness layer

A harness is only real if it changes behavior. The translated companion pattern changes behavior by giving the controller and the wiki a concrete test:
- can the same claim survive in multiple framings?
- can one framing remain executable and status-disciplined?
- can nominalist wording be enforced without losing technical content?

This page exists so those questions have one current, inspectable yes-case.
