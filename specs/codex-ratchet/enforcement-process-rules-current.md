## Hard Build Guardrail

The current mandatory stage order is:

1. keep tool sims active until the needed tool surface is honestly covered;
2. split tool sims into micro-probes: one tool, one function/API surface, one tiny claim, one positive test, one negative test, one boundary test;
3. make each tool find bounded legos to test itself on before using that tool inside lego-stage claims;
4. keep tool-integration sims active only after the individual coupled tool functions have receipts;
5. keep lego sims active across the registry, one bounded lego at a time;
6. use parallel workers inside bounded packets without widening the stage;
7. treat classical baselines and controls as baseline/control evidence only;
8. run coupling/coexistence after lego-stage completion by default;
9. keep earlier coupling exceptions explicit, bounded, exploratory, and routed back into tool/lego refinement;
10. block broad coupling/coexistence/topology/emergence promotion until parent evidence is strong enough;
11. keep bridge, axis, and engine surfaces late and gated.

This guardrail overrides any queue, prompt, worker plan, or narrative that widens exploratory coupling into earned higher-stage permission.

## Full Wizard Sim Guardrail

For sim work, Full Wizard is a max-useful parallel process, not a serial explanation format.

Tool-stage and lego-stage authoring/audit work should fan out across independent tool/function/API surfaces. Each worker still owns one bounded packet: one tool, one function/API surface, one tiny claim, one bounded fixture or lego target, one positive case, one negative case, one boundary case, and one demotion condition.

Runner seriality does not suppress LLM-worker parallelism. Git/index changes, queue writes, result commits, and status-label changes stay controller-serial.

## Current Plan Guardrail

The corrected operating plan remains:

1. start from the two root constraints;
2. build admissible carrier and probe math first;
3. build foundational legos as independent pure-math sims;
4. preserve the layered geometric-constraint-manifold view;
5. do not force canon on the current candidate order;
6. after relevant parent legos are current, allow bounded stack/composition tests without treating that as closure.

Density matrices, left/right Weyl spinors on nested Hopf tori, flux dependency chains, and Pauli/Bloch/channel/differential machinery should be treated as legos or candidate structures before compound claims.

## Contract Definitions

- `classical_baseline`: useful baseline/control evidence, not a target nonclassical substrate.
- `canonical sim` / `canonical by process`: fresh rerun plus SIM_TEMPLATE-style structure, `classification`, non-empty tool manifest reasons, and claim-relevant load-bearing tool depth. This does not by itself mean `nonclassical`.
- `micro tool sim`: smallest tool-stage probe against one function/API surface and one tiny claim.
- `tool-lego fit probe`: pre-lego tool-stage evidence that a tool can carry a bounded lego-shaped question. It does not promote the lego.
- `tool-tool coupling`: tool-stage integration only after the exact participating tool functions have receipts.

## Tool Depth Rule

Before a tool is load-bearing inside lego-stage or integration claims, the relevant function/API surface needs a micro receipt with:

- one named tool;
- one named function/API surface;
- one tiny claim;
- one bounded lego target or fixture;
- positive, negative, and boundary behavior;
- a demotion condition.

Runner `DONE` is execution evidence only. Controller admission requires reconciling queue row, result JSON, `classification`, claim-relevant `TOOL_INTEGRATION_DEPTH`, and ledger loopback.

## Anti-Salience Rule

The repo names the structural failure mode directly: models get pulled toward novel claims, high-level patterns, bridge/engine framing, and polished stories. The required intercept is to cite the gate criterion and result file. If the file cannot be cited, the gate is not satisfied.

This mirror can guide repo work and wiki routing. It cannot promote a sim, validate a result, admit a bridge/axis/engine claim, or override the live repo source.
