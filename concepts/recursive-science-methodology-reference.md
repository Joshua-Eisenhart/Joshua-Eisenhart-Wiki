# Recursive Scientific Methodology Reference

## Overview
Leviathan's science method is not just “do experiments and keep the good ones.”
It is a recursive empirical discipline built around mirrors, accounting, feedback, and continuous correction.

This page extracts that method in cleaner operational terms.

## Role in the live wiki cluster
- **Strongest use:** current support page translating Leviathan's recursive mirror/accounting language into a reusable update discipline for CS/dev readers.
- **Weak use:** not the right page for live interface contracts, module ownership disputes, or broad worldview genealogy.
- **Authority boundary:** current method-translation surface. It is downstream of the legacy sources and upstream of the repo contract surfaces; it does not outrank `docs/specs/`, `docs/ARCHITECTURE.md`, or `docs/ROADMAP.md`.

## Core pattern
The recurring pattern is:
1. generate a candidate state, claim, or proposal
2. confront it with reality or a mirror surface
3. account for the result explicitly
4. update the next state from that result
5. repeat at the next scale

That is why the method is recursive.
It is not only repeated over time. It is repeated across levels:
- person
- organization
- ecosystem
- system architecture

## Why mirrors matter
In the Leviathan material, mirrors are not just self-help tools.
They are empirical interfaces.

A mirror does three things:
- exposes current state
- forces explicit accounting
- provides a basis for recursive correction

That is why the same pattern can apply at:
- individual scale
- organizational scale
- ecosystem scale

## Why this is more than governance
The science method is load-bearing because Leviathan does not separate:
- epistemic correction
- moral/accountability correction
- organizational correction

In the framework, these all use recursive feedback.
That is why the method can connect to:
- mirrors
- world models
- development loops
- QIT engine cycles
- prediction-first memory/perception

## Operational CS translation
In usable developer language, the recursive scientific method is a bounded update discipline:
- maintain explicit current state
- expose it to bounded evidence surface
- record the delta
- update the state
- rerun

That is much closer to:
- control loops
- state-transition systems
- audit-backed refinement
- recursive model correction

than to one-shot theorizing.

## JP / Lev repo crosswalk
The live repo maps this method to runtime labels like:
- candidate state / claim / proposal -> **PROPOSE** within the tick lifecycle
- mirror or evidence confrontation -> **OBSERVE** plus policy/evidence checks at **GATE**
- explicit accounting -> event-backed audit on the Event Bus
- state correction -> **APPLY / UPDATE** against explicit graph state
- repeated bounded iteration -> FlowMind policy plus Orchestration execution boundaries

That is why this page matters for JP-facing handoff: it translates the Leviathan mirror language into a runtime discipline that already has recognizable repo-native surfaces.
