# Support-First Constraint-Manifold Dependency Chain

## Overview
This page captures the broader thread-level correction that the wiki had not yet made explicit enough.

The system should not be read as operator-first or causality-first. It is support-first and constraint-first:
- first identify candidate support layers / carrier surfaces
- then identify what objects can run on those supports
- then identify what operators can act there
- then identify what couplings survive
- then bring the layers into simultaneous execution

The important relation is not just "is related to" but `runs on`.

The harness-side correction is that the manifold should also clean up default object-first and causality-first narration as the stack builds. Later layers should not silently re-import primitives that lower layers already rejected.

Harness-time implication:
- this page is not only a later explanatory note
- it is a second-layer controller doctrine surface that should be loaded early enough to block premature object-first, operator-first, and bridge-first narration

## Why this page was needed
Several existing pages already preserve pieces of this view:
- geometry before axis
- constraints before entropy
- admissibility before promotion
- classical baselines and negative sims as real evidence

But the wiki still did not clearly capture the full thread-level distinction that:
- support relations are candidate relations that must be simulated, not declared
- the ratchet behaves like a dependency chain, not just a generic filter
- classical processes can run inside the larger manifold rather than grounding it
- all legos, even classical and negative ones, can eventually sit on the same mature sim manifold at different dependency depths

## Support-first rather than operator-first
A lot of mathematical and physical language starts by making operators primary. This project inverts that.

The primary question is not:
- which operator is fundamental?

The primary questions are:
- what support / carrier / geometry is admissible?
- what can run on that support?
- which operators become admissible only after that support exists?

A good informal description from the thread was: operators need geometry the way functions need graph paper.

The same inversion applies to explanation:
- not "what caused this object to appear?"
- but "what distinctions survived, on what support, under which constraints, strongly enough to let the later layer exist at all?"

## `Runs on` is a sim question
For any pair of structures `X` and `Y`, the relation `X runs on Y` should be treated as a candidate sim object.

That means:
- it may be true that `X` runs on `Y`
- it may be false
- it may only be true in one narrow sense (representation, support, transport, operator domain, coexistence)
- the relation must be earned through sim and evidence rather than declared from naming similarity or analogy

So the system should not jump from:
- these structures feel close

to:
- one definitely runs on the other

without a direct bounded check.

The same applies to causal wording:
- do not jump from "this changes with that" to "this causes that"
- first ask whether the claimed distinction survives the active support/manifold/probe conditions
- only later, if a lane earns it, allow a narrower causal reading

The same applies to identity/equality/time/probability language:
- do not quietly let later pages treat them as primitives after lower layers rejected that move
- if the lane cannot show how those readouts are admitted on the active support, treat them as candidate language or shorthand only

## Ratchet as dependency chain
A useful current formulation is:
- a ratchet is an ordered dependency chain that builds mathematical structure from constraints by progressively allowing only what survives prior admissibility conditions

This has three aspects at once:
1. dependency: later layers require earlier surviving structure
2. constraint: not everything is admissible at each stage
3. survival / allowance: some structures are permitted through and become the substrate for later structure

So the ratchet is not just blind subtraction. It is constrained foundation-building.

## What this should enforce

### For harness behavior
- load support-first before heavy operator or bridge synthesis
- force later pages to answer what the support is, what runs there, and what evidence earns that relation
- block default object-first explanation when the dependency question is still open

### For controller/runtime behavior
- keep candidate-blind and fail-closed behavior where enforcement is the job
- do not promote a relation from suggestive similarity into admitted support just because the story reads well
- keep open stack order and coexistence questions open until bounded stack tests exist

### For sim design
- start from carrier/support/probe admissibility
- simulate geometry and coexistence early on the same carrier
- treat `runs on`, stack order, bridge relations, and coupling relations as bounded sim objects
- require negative controls and wrong-order variants so the ratchet is doing selection instead of confirmation

## Candidate-guided, not candidate-dogmatic
The search space is too large to explore without candidate structures. So the process needs:
- candidate topologies
- candidate layer orders
- candidate support relations
- candidate couplings

But candidates are search scaffolds, not truths by declaration. In the ratchet's perspective, a favored candidate has no special status until it survives constraint pressure better than alternatives.

This is why alternate models matter:
- nearby alternatives
- classical baselines
- negative sims
- wrong-order variants
- incompatible coupling variants

Without these, there is no real ratchet — only confirmation.

## Classical lives inside the manifold
A major thread correction was that the project should not be reframed as if it first needs to establish a classical lane.

The repo already has a large classical baseline set, especially through numpy sims. More importantly, the larger system hypothesis is:
- causal and classical processes can run inside the larger system
- the larger system does not run on the classical

So classical/numpy realizations should be treated as:
- real baseline occupants of the manifold
- comparison surfaces
- falsification/control surfaces
- not as an external competitor ontology

This yields a useful audit frame:
- numpy/classical realization
- richer QIT / constraint-stack realization
- same bounded object if possible
- compare what survives, collapses, or diverges

## All legos can eventually share one manifold
Long-term target framing only. No final manifold is established here; promotion
requires current repo receipts closing support/order/coexistence tests.

The long-term target is not a pile of forever-disconnected probes.

The thread-level picture is:
- all the legos, including classical and negative ones, can eventually be placed on one mature sim manifold
- each lego occupies a different dependency location
- some are broad supports, some are narrow later structures, some are controls, some are exclusions

So the final manifold is not just the winning construction. It is the structured space in which:
- admissible objects live at their proper depth
- excluded regions are still represented by graveyard/negative evidence
- classical realizations occupy a real subregion
- richer constraint/QIT realizations occupy narrower or later subregions

## Operators and entropy are later layer questions
The thread also clarified an ordering discipline:
- different layers may admit different operators
- different layers may admit different entropy types or readouts
- but this classification becomes honest only after the manifold layers and their support order are built and running together

So operator-family and entropy-family mapping should be treated as later classification work over the running manifold, not as the starting point.

And causal language should be treated the same way:
- a later classification or admissible readout if earned
- not a primitive assumption that gets smuggled into the manifold from the start
