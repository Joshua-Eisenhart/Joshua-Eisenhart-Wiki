# Model Context Overlay

## Overview
This page is a controlled overlay surface, not a canonical doctrine page.

Its value is that it shows how the older Eisenhart material tried to place itself relative to academic traditions, adjacent research programs, and broader worldview frames. Its risk is that it can smooth sharp distinctions away and normalize the system into a generic philosophical or agent-framework story.

So this page should be used as translation fuel and genealogy, not as primary authority.

## Role in the live wiki cluster
- **Strongest use:** legacy translation/genealogy surface for academic placement, worldview context, and adjacent-tradition lookup.
- **Weak use:** poor primary interface for dev-facing handoff, runtime ownership, or proposal-versus-earned-state decisions.
- **Authority boundary:** bounded interpretive overlay. Useful for translation cues and historical context, but it must not outrank source-bound corrections or live repo contracts.

For actual runtime structure, prefer the live Lev repo docs. For dev-facing wiki translation, prefer [[graph-driven-intent-runtime]], [[recursive-science-methodology-reference]], and [[holodeck-qit-fep-leviathan-integration]].

## Why it still matters
Even though this page is not authoritative, it preserves a useful historical fact: the older Leviathan lane was not just trying to make a tool. It was trying to fuse:
- semantic computing
- intent understanding
- context assembly
- adaptive collaboration
- worldview and governance framing

That genealogy matters for JP handoff because otherwise the social-OS and semantic-computing motivation can disappear and leave only a thin technical shell.

## Anti-drift rule
The raw source explicitly warns that this overlay may contain hallucinated or over-broad connections.

That means this page must never silently do the following:
- turn the system into a generic agent framework by analogy
- smooth contradictions away because a comparison sounds elegant
- let broad academic mappings outrank the system's own layer/process discipline
- treat interpretive overlay as stronger than source-bound or repo-bound authority

A good use of this page is: take a nearby tradition, extract the useful structure, then restate it in the wiki's own constrained language.

## What the overlay is good at
The strongest durable uses are:
- placing the system near QIT reconstruction, operationalism, process-first thinking, and anti-reification traditions
- showing why probe-relative identity, constraint survival, and translation discipline matter
- identifying thinkers whose work is structurally adjacent but still carries classical or metaphysical residue
- explaining why the wiki keeps building a nominalist CS translation layer instead of simply copying academic language

## JP / Lev runtime crosswalk
For JP-facing handoff, this page should be read as genealogy behind the runtime, not as the runtime itself.

The live Lev repo already makes several things explicit in cleaner engineering language:
- policy/control is separated from execution
- graph state is separated from event causality
- contracts and ownership boundaries outrank loose explanation
- the runtime is not just a chat shell

So the best use of this page is to explain why the project cares about intent, context, and semantic computing so much, while letting the repo docs define the actual engineering boundary.
