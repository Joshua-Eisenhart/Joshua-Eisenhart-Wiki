## Tool roles

### XGI
Best role:
- hypergraph and multi-way relation carrier;
- higher-order incidence;
- non-pairwise coupling witness;
- layered relation structure.

Use when pairwise graph edges would destroy the claim.

Bad use:
- using hypergraph language while still encoding only pairwise relations.

### TopoNetX
Best role:
- simplicial and cell-complex structure;
- boundary operators;
- higher-order adjacency/incidence;
- carrier spaces where cells, not only nodes/edges, matter.

Use when the relevant relation is a complex with faces/cells and boundary maps.

Bad use:
- claiming cell-complex evidence from a graph-only probe.

### GUDHI
Best role:
- persistent homology;
- filtrations;
- topological signatures and stability under scale;
- whether a candidate basin has durable holes/components/features.

Use when a topology claim needs persistence across thresholds or scales.

Bad use:
- treating a persistence signature as full ontology closure.

## Basin / systems-philosophy connection
Topology tools are especially important for the attractor-basin inversion.

A basin should not be named first. It should be inferred after wide trajectories and probes repeatedly show durable structure.

Tool mapping:
- XGI asks whether the basin is multi-way rather than pairwise.
- TopoNetX asks whether the basin has cell/boundary structure.
- GUDHI asks which features persist across filtration scale.

This connects directly to [[systems-philosophy-attractor-basin-inversion]].

## Status discipline
Use these labels carefully:

- `present`: the tool appears in code/docs/results.
- `capability`: a bounded probe shows the tool can return expected object types.
- `supportive`: the tool helps inspect a claim but does not decide it.
- `load_bearing`: the returned topological object changes admission/exclusion.
- `canonical by process`: only when the repo contract admits it with current evidence.
