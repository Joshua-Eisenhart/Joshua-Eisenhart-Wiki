# Tradition-System Mapping (Detailed)

## Overview
Date: 2026-04-05. Status: DRAFT. Maps external formal traditions to the system's specific claims. Each mapping should be audited for smoothing, forced fit, or overclaiming. These are STRUCTURAL CORRESPONDENCES, not identities or proofs.

## Constraint on Distinguishability <-> Formal Concepts
- Constraint on distinguishability <-> Operational equivalence (Spekkens): Strong fit. Mismatch: Spekkens is within QM ontological models; system is broader.
- a=a iff a~b <-> Identity of indiscernibles (Leibniz PII): Strong. PII is philosophical; system makes it operational via finite probes.
- Constraint surface M(C) <-> Quotient structure S/~_M: Strong. System adds F01 (finite probes) and N01 (composition order).
- Probes are finite (F01) <-> Coarse-graining: Moderate. Coarse-graining is a choice; F01 is a constraint on what exists.
- Processing can't increase distinguishability <-> Data processing inequality: Strong. DPI is a theorem; system treats it as structural.
- Geometry of distinguishability <-> Fisher-Rao metric (Cencov): Strong. System may need quantum version (Petz: non-unique).

## Survival Under Constraint <-> Viability Theory
- What survives on M(C) <-> Viability kernel (Aubin): Strong. Viability is for controlled systems; system is constraint satisfaction.
- Not pulled to an attractor <-> Viability != attractor basin: CRITICAL. System explicitly rejects attractor-pull language.


Live repo evidence now makes this split concrete rather than rhetorical. `sim_viability_vs_attractor.py` contrasts admissible survival against attractor collapse on the same bounded carrier, `sim_axis0_attractor_basin_boundary.py` shows a separate basin-boundary/process-specific question, and `sim_qit_attractor_basin_recovery.py` earns a narrower equivalence-class recovery claim under ordered noncommuting channels. So viability remains the broader survival primitive; basin language has to be earned for specific ordered families.
