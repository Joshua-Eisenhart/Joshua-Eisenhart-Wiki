# Mimetic Meme Manifold Harness

## Overview
The language layer that makes the harness work. About salience-shaping, not decorative wording. A harness needs a dictionary of aligned language that makes the right concepts salient to the model at boot. LLMs follow salience gradients — if the language field is wrong, the model defaults to generic classical smoothing.

## What It Is

The mimetic meme manifold is the macro arena in which many engine instances couple, imitate, synchronize, fork, and prune through language/meme dynamics (Girard mimesis + Dawkins memetics). It is an engine-of-engines layer, not just a per-terminal prompt layer.

## The Language Stack

| Layer | Content |
|---|---|
| 1. Root constraints | constraint on distinguishability, finitude, noncommutation, probe-relative identity |
| 2. Harness grammar | survived/killed/open/not_yet_tested, probe family, admissible geometry, carrier, witness, negative control, promotion status |
| 3. Project dictionary | project-specific terms that bias model toward current goal; must stay consistent with root constraints |
| 4. Boot prompt templates | A1 recon, B ratchet, A0 compiler, A2 mining, SIM discipline — each has different salience profile |

## Why This Matters

LLMs do not merely parse meaning — they follow salience gradients. If the language field is wrong, the model defaults to generic classical smoothing. The harness-language dictionary is not cosmetic — it is part of the control system. Whoever controls the dictionary controls what becomes salient, what gets ignored, and what the model treats as real.

## Relation to Leviathan

Constraint engineering and Leviathan-language control are tightly linked because both shape admissibility, constrain what can be seen as relevant, maintain separation of boots and duties, prevent contamination across roles, and make the right structure more likely to survive. The language manifold is one of the control surfaces of the harness.

## How to Build It

1. Start from root constraints
2. Extract actual terms already used in repo
3. Separate reference vocabulary from operational vocabulary
4. Write boot prompts that bias salience without inventing new ontology
5. Test boot prompts with actual runs
6. Keep dictionary versioned and auditable
