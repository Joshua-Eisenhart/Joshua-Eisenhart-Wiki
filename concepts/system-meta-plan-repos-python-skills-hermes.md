# System Meta Plan -- Repos, Python, Skills, Hermes

## Overview
Broad umbrella planning surface for the whole system setup. Narrower docs outrank it when more specific. Intentionally broader than the nonclassical tool-plan doc. Master planning surface to audit and improve.

## Governing Principles
1. Do planning before mass installs, tests, or deletions
2. Do not ask Hermes to read the whole system at once
3. Feed Hermes bounded packs slowly, with explicit roles
4. Do not let underpowered sims count as real closure
5. Do not delete old environments until replacements verified
6. Keep Git repos in ~/GitHub
7. Keep main repo on Desktop
8. Standardize Python/tooling first, then clean up redundancy
9. Only keep a tool if it has a named job in the pipeline
10. Hermes is not just a tool -- it is part of the architecture

## Core Strategic Direction
Immediate priority: get QIT engine prototype ratcheted properly, get real pre-Axis sims working, make those sims honest and scientifically respectable. Broader system should later inherit the shape of QIT engines. System graphs and architecture should reflect the engine and its constraints.

## Hermes Priority
Hermes is effectively the A2 high-entropy ingestion plane. Hermes itself, its add-ons, and its improvement surfaces are first-class planning objects. See [[hermes-stack-and-addons-plan]].

## Stack Categories
1. Scientific substrate (QIT engine, pre-Axis sims, constraints)
2. Architecture (host planes, runtime kernel, enforcement)
3. Tools (install, verify, wire, classify)
4. Hermes stack (core, improvements, ecosystem, skills)
5. Cleanup/reorganization (Python, repos, skills, docs)

## What This Plan Does NOT Do
Does not declare canon. Does not replace narrower docs. Does not authorize mass changes. Does not claim closure on any open scientific question.
