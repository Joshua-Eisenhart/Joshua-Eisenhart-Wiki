# Legacy Source History

## Overview
Historical source material: copies, bootpacks, digests, and provenance notes. Useful for origin tracing, but not a current/front-door authority layer by itself.

Legacy material can still preserve strong candidate structure; age alone does not demote a claim. This page routes lineage and provenance, not authority.

Use this page as the entrypoint for legacy source strata, not as the place to learn the current doctrine directly.

## Routing by lineage
When tracing the overlapping legacy cluster, use the source history in this order:
- framing boundary -> [[current-docs-vs-legacy-framing]]
- governance / social-OS lineage -> [[leviathan-framework]] and [[legacy-governance-bootpacks]]
- memory / world-model lineage -> [[holodeck-docs]] then [[projective-holodeck-memory-model]]
- personality / engine-grammar lineage -> [[legacy-psychology-personality]] then [[emotional-evolution-personality-system]] then [[personality-theory-mapping]]
- foundational synthesis / cosmology lineage -> [[grandmaster-of-the-universe]] and [[the-dark-empress]]
- intent / worldview framing lineage -> [[a2-intent-summary]] and [[model-context-overlay]]

Use [[legacy-speculative-frameworks]] to decide which major legacy branch a page belongs to before reading it as doctrine, and use [[dark-empress-vs-grandmaster]] when the question is which of the two major legacy books is actually carrying the claim's center of gravity.

Use [[legacy-speculative-theory]] as a warning-label surface when a source starts fusing cosmology, governance, personality, and world-model language into one undifferentiated package.

For the holodeck lane, hand CS/dev readers forward from the legacy surfaces into [[holodeck-as-recall-space]], [[prediction-first-memory-vs-llm-memory]], and [[holodeck-qit-fep-leviathan-integration]] rather than treating the raw legacy source as a settled spec.

For the personality lane, treat MBTI/Jung-heavy mapping pages as genealogy and candidate engine-grammar carriers, not as the primary dev-facing explanation. Hand forward into [[leviathan-science-method-qit-engine-crosswalk]] when the reader needs the current synthesis/translation surface.

For the Leviathan OS / JP-facing lane, use [[leviathan-framework]] and [[model-context-overlay]] as genealogy, then hand forward into [[recursive-science-methodology-reference]], [[graph-driven-intent-runtime]], and [[holodeck-qit-fep-leviathan-integration]] for the cleaner runtime and world-model rendering.

That keeps provenance, extracted kernel, and current synthesis from collapsing into one layer.

## Role in the live wiki cluster
- **Strongest use:** provenance-first entrypoint for the overlapping legacy source families before any branch is treated as doctrine.
- **Weak use:** not the right page for deciding which legacy framework dominates, for book-vs-book comparison, or for current dev-facing translation.
- **Authority boundary:** legacy routing/provenance surface. It preserves source strata and handoff direction; it does not itself settle present-tense truth claims.

## Recommended reading order
1. Read [[current-docs-vs-legacy-framing]] first if the reader still needs the current-versus-legacy boundary.
2. Read this page next to identify which legacy family or source stratum is in play.
3. Continue to [[legacy-speculative-frameworks]] to separate the major branch families.
4. Use [[dark-empress-vs-grandmaster]] only when the ambiguity is between the two major legacy books.
5. Use [[legacy-speculative-theory]] if a source starts blending too many branches at once and needs a warning-label surface.
6. Then hand forward into the specific branch page or current bridge page the question actually needs.
