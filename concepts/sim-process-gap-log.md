## What the source says, safely

The source is a derived audit surface. Its own status says it is for diagnosis and next-gap visibility, not the primary live queue.

The main signal is not "nothing works." The signal is narrower and more useful:

1. many local result anchors are real or strong;
2. promotion pressure and status-label discipline remain live gaps;
3. older strong files may need modern process framing before stronger labels;
4. sparse tool lanes can have one good anchor without being broadly integrated;
5. higher-stage volume does not equal earned higher-stage permission.

That keeps two live readings separate:

- **Positive reading:** the estate has meaningful local anchors in geometry, graph/topology, proof, equivariance, and classical/QIT-adjacent lanes.
- **Caution reading:** those anchors do not automatically admit bridge, axis, engine, manifold, or canonical-by-process claims.

## Gap groups from the repo source

### Promotion and conformance debt

The source names promotion pressure as the main open process gap. It also records canonical-conformance debt: a prior 2026-04-18 read-only scan found `634` canonical-labeled probes, `571` conformant, and `63` violating current packet rules.

Claim ceiling: this page did not rerun that scan. Treat those numbers as source-reported prior-audit evidence until a fresh audit repeats them.

### Strong older anchors needing modern framing

The source says some older geometry and baseline surfaces may have good math quality while still lacking the modern process-honest shape needed for stronger public labels. Examples named in the source include Hopf/geomstats/Clifford geometry, Carnot/Szilard baselines, shell-local Weyl, graph topology, reduced-state, and negativity-measure families.

Use this as a normalization queue, not as promotion.

### Tool lanes with narrow but real anchors

The source separates narrow anchor evidence from broad lane completion for TopoNetX, PyG, cvc5, and e3nn.

Safe reading:
- a named anchor can be real;
- one anchor does not make a whole tool lane complete;
- each tool/function/API surface still needs scoped receipts before being load-bearing in downstream lego, coupling, bridge, or engine claims.

### Debugging and runtime hygiene lanes

The source records process/debugging lanes around partial-trace correctness, optional decorative `clifford` import failures, and probe-local cache setup.

Safe reading: these are useful maintenance/debugging signals. They should lead to targeted confirm/fix passes, not blanket demotion or broad status changes from the audit summary alone.

### Higher-stage inflation and boundary rebuild debt

The source says higher-stage pairwise/coexistence/bridge families are inflated and that Tier D boundary work is broken on substance, not only polish, because current D1-D4 probes use toy integer/proxy encodings rather than real lower-layer structure.

Safe reading: the classical/QIT lane is real but narrow. The next honest move is bounded re-audit, not broader engine promotion.
