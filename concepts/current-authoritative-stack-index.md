# Current Authoritative Stack Index

## Overview
Historical `system_v4` owner-stack mirror with current-v5 overlay. This page
exists so the wiki has a public route for older owner-language routing surfaces,
but live status belongs to current `system_v5` repo/spec mirrors.

## Current v5 overlay
For live sim, tool-role, manifold, provider-receipt, and formal-scout status, use the current `system_v5` spec mirrors before reading the older `system_v4` owner-stack mirror. Concept routers remain useful as historical/router pointers.

- [[specs/codex-ratchet/sim-estate-integration-status]] is the current geometric-constraint-manifold and sim-estate status mirror.
- [[specs/codex-ratchet/formal-scout-readiness-status]] is the current formal-scout readiness and validator-health status mirror.
- [[specs/codex-ratchet/tool-function-receipt-status]] is the current tool/function receipt status mirror.
- [[axis0-current-doctrine-state-card]] carries a current v5 delta: Axis0 remains `open_partial`, not solved.

The `system_v4` owner stack remains useful genealogy and owner-language doctrine, but it must not override current `system_v5` receipt boundaries, `promotion_allowed=false`, tool quarantine, or the `grok_sim` proposal-only fence.

## What it controls
- Controller authority stays with `THREAD_CONSOLIDATION_CONTROLLER.md` and its compact state companions.
- The current Axis 0 owner stack is summarized through [[axis0-current-doctrine-state-card]] and [[constraint-geometry-axis0-separation]].
- Thread B remains a staging/support lane, not a doctrine lane; see [[thread-b-stack-audit]].
- Live sim admission and validator status are not controlled here; use the current v5 spec mirrors above.

## Current routing rule
Read this page before using broad `system_v4/docs` clusters as stronger owner-language surfaces. For live sim status, read the current v5 spec mirrors first. The repo’s own rule is narrow:
- owner packets first
- support packets second
- superseded packets never silently promoted back into live routing

## What not to do

- Do not treat superseded runbooks as active doctrine.
- Do not let the wiki’s older mixed-framing pages outrank this repo-current routing surface when the question is specifically about the live `system_v4` stack.
- Do not let this `system_v4` stack mirror outrank current `system_v5` formal-scout/readiness/sim-estate evidence when the question is live sim status.
