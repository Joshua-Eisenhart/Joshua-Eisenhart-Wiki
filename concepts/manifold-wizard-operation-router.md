# Manifold Wizard Operation Router

## Status boundary
This page is a wiki router for a repo operation note. It records that the repo artifact exists and gives future workers a stable reading path. It does not admit, promote, or canonicalize any manifold, G-structure, physics, bridge, axis, engine, or target-system claim.

It is also not a Wizard run receipt. For sim, proof, tool-stage, lego, queue, or result work, current process routing goes through the repo authority files and `system_v5/ops/SIM_FULL_WIZARD_PARALLEL_RUNBOOK.md`, then through current status mirrors such as [[specs/codex-ratchet/formal-scout-readiness-status|formal-scout-readiness-status]] and [[specs/codex-ratchet/sim-estate-integration-status|sim-estate-integration-status]].

## Repo Operation Target
The repo operation note frames the working target as the geometric constraint manifold: nested simultaneous constraint shells with order-sensitive geometry, not axis labels and not only the Weyl/chirality layer.

2026-07-01 carrier correction: PEPS/PEPS3D language in older operation notes is stale. The retired path is the single-fused PEPS3D contraction; CTMRG remains usable inside nested 2D PEPSKit layers when contraction-error certificates bound the readout. Current carrier routing is nested PEPS2D/Hopfield connection geometry, ITensors-MPS, exact dense/TensorKit, QuantumClifford, and spinor-native trajectories, with JAX and Julia primary in parallel. Axis0, attractor-basin receipts, graph/proof tools, auto_LiRPA, and le-wm remain evidence surfaces over that target in the repo operation note. Treat this as process routing, not independent wiki canon; future wiki/repo work should not collapse the manifold program into axis labels or a Weyl/chirality-only story.

## MMM loading rule
Current manifold-provider lanes use Wizard v4.2. A provider receipt only counts as MMM-loaded when it records `wizard_mmm.mmm_loaded=true`, source paths, digests, route card, council role, mini-MMM ids, and prompt hash.

MMM-informed prose is useful, but it is not the same as an MMM-loaded council lane.

## Practical council standard
For external manifold council lanes, the repo note asks for a real model call, compact MMM in the prompt, relevant mini-MMM slices for specialized lanes, a distinct role/route card, a durable provider receipt, and a concrete finding that changes, kills, blocks, or sharpens the next move.

## Loop shape
Keep these lanes separate:

1. Decision: smallest manifold-rooted repair.
2. Failure: premortem/falsifier when scout status could be mistaken for promotion.
3. Tool: one tool/function/manifold role per packet.
4. Graph/proof: z3/cvc5, rustworkx, PyG, XGI, TopoNetX, GUDHI, and related proof/graph tools against non-tautological predicates.
5. Tensor/carrier: ITensors-MPS, exact dense/TensorKit, QuantumClifford, spinor trajectories, JAX, Julia, PyTorch/autograd, auto_LiRPA, and le-wm must carry branch-dependent signal. PyTorch is legacy/comparison/helper unless a current receipt bounds its role.
6. Compile: killed/open claims, next receipt, validator, stop condition, and claim ceiling.

## Stop conditions
Stop or shrink when row counts are mistaken for manifold depth, named layers are index-driven rather than semantically implemented, proof predicates survive only by label, le-wm/auto_LiRPA do not discriminate branch-dependent data, Axis0 scalar projection is treated as repaired without multidimensional actuation, or provider output is cited as evidence instead of proposal/falsifier source.
