# Prompt Memory Intention Formalization Router — 2026-06-17

## Purpose

This page is the receiving router for Josh's observation that many Claude, Codex, and Hermes memories contain prompt-like material that preserves deeper intentions and ideas.

The goal is not to dump those memories into the wiki. The goal is to process them into formal wiki surfaces with:

- provenance fences;
- owner-kernel versus assistant-elaboration separation;
- research support lanes;
- explicit claim ceilings;
- live alternatives kept open until bounded tests exclude them.

This router is a workbench. It does not admit a theorem, a final `M(C)`, a QIT engine, Axis0, bridge physics, or canonical doctrine.

## Source classes and authority status

| Class | Example surfaces | Authorship / authority status | Allowed use | Main risk / guard |
|---|---|---|---|---|
| Raw owner prompt/correction | `raw/owner-prompts/*`, explicit copied owner prompt windows, owner-confirmed corrections | potential owner-kernel only when exact quote or owner confirmation is present | provenance ledger, route patch, concept-page note with quote window | do not generalize beyond the quoted window |
| Agent memory reservoir — Hermes | private Hermes compact memory | mixed/private steering memory; may contain owner preferences plus assistant compression | candidate pointer only; must be re-ledgered before citation | too compressed to be evidence by itself; never quote unrelated private profile material |
| Agent memory reservoir — Claude | private Claude project memory / row files | mixed owner instructions, assistant summaries, and generated synthesis | candidate pointer only; use for finding possible owner-corrections | generated synthesis can sound like owner doctrine |
| Agent memory reservoir — Codex / Codex-second | private Codex memory, summaries, chronicles, rollout notes | mixed run-state, assistant-generated summaries, owner corrections, repo pointers | candidate pointer only; validate against repo/wiki receipts | stale operational state can masquerade as durable intention |
| Existing wiki source/intake | `raw/owner-prompts/*`, `raw/articles/*`, project/concept pages | authority varies by page status and evidence label | cite with page status and claim ceiling | raw source or wiki existence is not automatically doctrine |
| Repo/wiki receipt | validators, sim receipts, exact project pages | evidence only at stated status | evidence anchor with status label | does not establish owner intention by itself |
| External comparator | papers/books/frameworks | external research comparator | support/falsifier lane only | must not be collapsed into owner ontology |

## First inventory result

The useful prompt-memory reservoirs are uneven:

- **Claude memory** has the densest owner-correction and prompt-intention material, especially project memory indices.
- **Codex memory** has the largest operational chronicle layer and many repo-state correction surfaces.
- **Codex-second memory** has compact foundation-language and installed-skill/toolchain correction surfaces.
- **Hermes memory** is intentionally compact; it should mostly point into wiki pages and skills, not carry the formal model by itself.

The first formalization pass should therefore treat memory as an **index and provenance map**, not as the final statement.

## Processing rule

Every candidate idea from memory must be split before it becomes wiki prose:

```text
memory row or prompt fragment
-> owner kernel / owner uncertainty / owner correction
-> assistant-generated elaboration
-> current repo or wiki evidence anchor
-> external research support or comparator
-> claim ceiling
-> next finite test or research tranche
```

If the split cannot be made, the fragment stays in a source/intake queue.

## Formalization output shapes

Use these output shapes instead of raw memory dumps:

1. **Candidate owner-kernel source ledger** — small table of owner prompt/correction fragments or owner-confirmed paraphrases. Required columns: `ledger_id`, `source_class`, `opaque_private_source_ref` or public wiki path, `line_or_char_window`, `quoted_window_or_redacted_paraphrase`, `speaker/provenance` (`owner` / `assistant` / `mixed` / `unknown`), `owner-kernel candidate`, `owner uncertainty/correction`, `assistant-elaboration risk`, `repo/wiki evidence anchor`, `privacy/secret scan result`, `redactions`, `target page`, `claim ceiling`, and `promotion state`.
2. **Research comparator page** — external/formal research lanes that support or challenge the owner-kernel without collapsing into it.
3. **Project router patch** — compact addition to the Codex Ratchet front door if the idea changes what future workers must load.
4. **Concept page deepening** — expand existing pages when the idea already has a concept home.
5. **Claim-ceiling receipt** — explicit note when a memory idea is important but not yet admitted.

Avoid one giant synthesis page. The model is too high-dimensional for a single winner-story.
