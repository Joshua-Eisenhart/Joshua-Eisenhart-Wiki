## Bottom line

The wiki already contains a mature harness doctrine. The current external research does **not** replace it; it sharpens the same direction:

```text
Harness = executable runtime boundary around the model.
```

The wiki's own version is stricter:

```text
Harness = current-authority routing + salience loading + receipt truth + execution gates + memory/restart sync + worker role separation + status discipline.
```

The strongest synthesis is:

```text
A good harness does not merely tell an LLM what to do.
It controls what counts as current, what may be read, what may be changed, what evidence upgrades status, which workers may act, and what gets preserved for the next thread.
```

## Unified local-wiki harness model

The local wiki's harness model can be expressed as a ten-part runtime:

```text
1. Front-door router
   reads current before legacy; selects project/system authority.

2. Source authority gate
   distinguishes source, archive, memory, current plan, runtime receipt.

3. Salience / MMM loader
   shapes behavior before reasoning/output.

4. Translation layer
   rewrites away from LLM defaults: reification, smoothing, causal story, fake closure.

5. Skill/procedure layer
   supplies reusable methods; should become executable when possible.

6. Action realization layer
   turns model intent into checked tool/action requests.

7. Receipt and route-truth layer
   proves what ran, what did not, and what status was earned.

8. Worker orchestration layer
   separates builder, runner, auditor, scout, falsifier, and synthesizer.

9. Memory/restart layer
   keeps repo/wiki/session/durable memory synced so fresh threads can restart.

10. Regression and improvement layer
    mines failures, proposes bounded harness edits, and validates before adoption.
```

## What to build from this

The next useful harness artifact is not another prose rule. It is an executable **runtime/action gate** that implements what the wiki already says.

Minimum contract:

```yaml
active_process_contract:
  project: <current project>
  system: <Hermes|Claude|Codex|other>
  authority_scope: <audit_only|write_allowed|sim_control|repo_maintenance>
  current_front_door: <wiki path>
  allowed_roots: []
  protected_roots: []
  required_skills: []
  allowed_actions: []
  blocked_actions: []
  evidence_required_for_status_upgrade: true
  route_truth_required: true
  exact_pathspec_required_for_git: true
```

Protected actions:

- writes / patches
- terminal mutating commands
- git add / commit / reset
- cron/background spawning
- subagent/workflow spawning
- cross-system file edits
- wiki front-door edits
- memory promotion

Core behavior:

```text
LLM proposes action → harness validates contract → harness executes or blocks → receipt written → memory/wiki updated if valid.
```

## Gaps found in the wiki corpus

1. **Runtime proof remains weaker than shape proof**
   - The wiki has strong receipt schemas, but many validators still prove artifact shape, not that a route truly ran.

2. **Executable skill layer is incomplete**
   - HASP-style Program Functions map naturally to this system, but most skills are still text procedures.

3. **Cross-system non-interference needs executable enforcement**
   - The doctrine exists; a runtime gate should block unauthorized edits across Hermes/Claude/Codex surfaces.

4. **Large append-only logs need compression into ledgers**
   - `_steward_questions.md` shows why raw blocker append logs become unusable.

5. **Memory promotion gates need to be explicit**
   - The wiki has restart/memory structure, but promotion from session to doctrine needs a stronger executable checklist.

6. **Harness edit protocol should be wired to tests**
   - The docs say harness edits need bounded scope + measurement; the next step is an always-run regression suite.
