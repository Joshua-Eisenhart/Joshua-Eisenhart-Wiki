# Leviathan Claim Ceilings

This page is the wording guard for Leviathan wiki pages.

## Claim Ladder

| Ceiling | Meaning | Safe phrase |
|---|---|---|
| narrative | product/category/story language | product narrative |
| provenance | user or source-origin note | provenance-only / user-provided |
| contract | spec/doc declares behavior | contract-backed |
| package | package, manifest, export, or script exists | package-backed |
| source | code implements a narrow behavior | implemented narrowly |
| test | test exists or can run | test-surface-backed |
| proof receipt | fresh command receipt at SHA | proof-backed at packet/SHA |
| product flow | user/operator path works end to end | product-flow-backed |
| release | install/docs/security/smoke/community gates green | release-ready |

## Red Claims

Do not use these unless a current proof packet establishes them exactly:

- fully green
- FEP runtime
- enterprise-ready
- launch-ready
- all plugins work
- world-model gates production decisions
- active inference is implemented
- AgentPing is production-ready
- AgentGuard is fully enforced
- default daemon proof is settled
- Codex Ratchet came from Leviathan
- JP originated Ratchet root constraints

## Yellow Claims

Use only with an evidence label:

- works
- implemented
- production
- default
- certified
- universal
- every surface
- every agent
- OS

## Preferred Wording

- source-backed
- contract-backed
- package-backed
- implemented narrowly
- proof-backed at packet/SHA
- split verdict
- design-backed
- provenance-only
- product direction
- needs current proof packet
