## 3. Single highest-risk OPEN finding

**Two findings are tied for highest-risk and neither subsumes the other — naming both, not collapsing.**

**Finding A — forgeable `sourceHashes`/`inputHash`/`graphStateHash` on engine obligations (Slice A).**
Strongest evidence: the `regression_reachability` skeptic didn't just read code — it wrote and ran a live vitest test calling `runClaimGateWaveRun` end-to-end with the **real, non-stub** `createRealThreeEngineObligationRunner()`, fabricated `sourceHashes`/`inputHash` from `Math.random()`-seeded strings with zero relation to real content, and got `"status": "passed", "findings": []`, with the fabricated hashes never appearing anywhere downstream after the format check. That upgrades the `obligation_hash` lens's claim from "never cross-referenced" to the skeptic's sharper finding: these fields are "fully inert decoration" — the real digest comes entirely from host filesystem/process state, completely independent of what the caller declared. Independently, the `obligation_hash` skeptic ran its own fresh, self-generated forged `sourceHashes` against the live CLI and got the same `passed`/`accepted` result, with the only correction being that "any consumer" overstates blast radius (zero current consumers of the proposal exist beyond the proof module and CLI passthrough). **This is forgery-class, not availability-class** — a caller-controlled value passes as if verified.

**Finding B — cross-process secret gap blocks the recursive loop (`graph_apply_chain`, `cross_process_module`).**
Strongest evidence: the `graph_apply_chain` skeptic minted a genuine, correctly-MAC'd ledger row in process A (`apply-graph-patch`, real `graphMutationApplied: true`), then fed that same real row into a fresh `record-graph-execution` invocation (process B) and got `status: graph_patch_blocked`, `graph_execution.graph_event_untrusted`, exit 1 — reproduced twice, deterministic. Root cause independently confirmed by direct code read: `HOST_GRAPH_EVENT_TRUST_PROCESS_SECRET = randomBytes(32)` is a bare module-level constant, no file, no env, no IPC (`core/orchestration/src/proof/claim-gate-loop.ts:1208` per skeptic's own re-derived line numbers — note the lens's original citations 615-630/743-745 were stale/from a different file copy, a sourcing-hygiene defect the skeptic flagged but did not treat as refuting the finding). The `cross_process_module` lens+skeptic pair independently confirms the fix exists (`host-trust-secret.ts`, file-sealed, race-safe, genuinely tested across a real spawned OS process) but is wired into zero production call sites — three separate `randomBytes(32)` instances remain live (`claim-gate-loop.ts:76`, `proof/claim-gate-loop.ts:1208`, `claim-gate-steering-run.ts:32`). **This is availability-class, not forgery-class** — no attacker gets a free mutation; the system fails closed, but it also fails closed for the legitimate multi-step chain, which is the architecture the whole spine is meant to support.

A third, lower-confidence finding worth holding alongside these, surfaced only by a skeptic, not its own lens: `seed_and_forgery_live`'s skeptic found that `runClaimGateWaveRun` itself has no gate at the function boundary — the project's own test (`claim-gate-wave-runner.test.ts:935-943`) calls it directly with an arbitrary literal `receiptRef` string, no MAC, and gets it echoed straight into `nextRunSeed`. It's currently inert only because of a `package.json` exports whitelist (confirmed via a live cross-package import attempt that threw `ERR_PACKAGE_PATH_NOT_EXPORTED`), not because the function refuses untrusted input. This is the same boundary-trusts-caller-token class as Finding A, one layer down, latent rather than active.

## 4. Are the three engines load-bearing or decorative?

**Load-bearing — strongest evidence in the whole audit set.** Not just "ran without erroring" — the `real_engines` skeptic did the one test that actually discriminates real-vs-decorative: live-tampered the actual Julia carrier file on disk (appended one comment line), reran `--real-engines` against the tampered file, and got a real failure (`exit 1`, `sourceHash must match expectedSourceHash`), then restored the file and confirmed the hash matched the original pin again. That rules out "the check exists in code but nothing actually re-hashes the live file." Independently: the lens measured 23.65s wall time for real subprocess spawns (`spawnSync` over `/opt/homebrew/bin/julia` and a real sim-stack `python3` with jax/torch/quimb/geomstats installed), and the skeptic found something the lens didn't even claim — the `threeEngineEnvelopeHash` **differed** between two identical-input runs while the input-derived `obligationHash` stayed constant, which is stronger anti-memoization evidence than wall-clock timing alone (a cached/decorative path would reproduce an identical envelope hash). Negative controls are a genuinely different code branch (`LEV_R0_NEGATIVE_CONTROL` env var), with `outputDiffersFromPrimary` asserted true in a live run. `commandBindsSourceRef` checks the exact hashed file path appears in the spawned process's actual `args`, closing the "hash one file, execute another" gap. **Caveat both lens and skeptic name identically and without softening: no CI test exercises this end-to-end — every test in the suite stubs the envelope builder. The "real" claim rests on live reproduction this session, status label `passes local rerun`, not `canonical by process`.**
